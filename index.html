The proposed spec is NOT about variables and I seriously wonder if we should not change the title of the document. You may call the feature it introduces "variables" but at the deeper level, that's not about variables.
Daniel Glazman, CSSWG Co-Chair, August 2012

[CSS Cascading Variables is] a family of custom author-defined properties known collectively as custom properties, which allow an author to assign arbitrary values to a property with an author-chosen name, and variables, which allow an author to then use those values in other properties elsewhere in the document.

First requested in 1998, first W3C proposal: 2008, first public draft April 2012

Nesting Rules

CSS

section { margin: 10px; } section nav { height: 25px; } section nav a, section nav a:visited { color: #0982C1; } section nav a:hover, section nav a:focus { text-decoration: underline; }

SASS

section { margin:10px; nav { height:25px; a, a:visited { color: #0982C1; &:hover, &:focus { text-decoration:underline; } } } }

Nesting is really useful for @media queries

Standard CSS

.some-class { /* Default styling */ } /* Hundreds of lines of CSS */ @media (max-width: 979px) { .some-class { /* Responsive styles */ } } @media (min-width: 980px) { .some-class { /* Desktop responsive styles */ } }

SASS

.some-class { /* Default styling */ @media (max-width: 979px) { /* Responsive styles */ } @media (min-width: 980px) { /* Desktop responsive styles */ } }

Clever @importing is the key

Bootstrap.less

// CSS Reset @import "reset.less"; // Core variables and mixins @import "variables.less"; // Modify this for custom colors, font-sizes, etc @import "mixins.less"; // Grid system and page structure @import "scaffolding.less"; @import "grid.less"; @import "layouts.less"; @import "media-grid.less"; // Base CSS @import "type.less"; @import "code.less"; @import "forms.less"; @import "tables.less"; // Components: common @import "sprites.less"; @import "dropdowns.less"; @import "wells.less"; @import "component-animations.less"; @import "close.less"; ...

Bootstrap's responsive.less

// REPEAT VARIABLES + MIXINS // ------------------------- // Required since we compile the responsive stuff separately @import "variables.less"; // Modify this for custom colors, font-sizes, etc @import "mixins.less"; // RESPONSIVE CLASSES @import "responsive-utilities.less"; // MEDIA QUERIES // UP TO LANDSCAPE PHONE @import "responsive-480px-max.less"; // Phones to portrait tablets and narrow desktops @import "responsive-767px-max.less"; // Tablets to regular desktops @import "responsive-768px-979px.less"; // Landscape tablets to large desktops @import "responsive-980px-1199px.less"; // Large desktops @import "responsive-1200px-min.less";

Modular CSS = DRY Code

Re-using CSS has always between projects has always been a challenge, tips and tricks from one project get lost and re-written for another. CSS pre-processors can change that

  • Want a responsive grid and a bunch of gradients? grid.less and mixins.less from Twitter Bootstrap and you're done. For Sass? Try Mooxins
  • Build yourself a pattern library like ClearLess by ClearLeft
  • How about all the gradients from Lea Verou's CSS3 pattern gallery? Download css-patterns-less

Minimal CSS

If you've got a page that only needs certain parts of the CSS, give them that stylesheet only. Mobile site? only give them the mobile styles but keep the vars and mixins

Going deeper

Extensible classes

Less

.module-a { /* A bunch of stuff */ } .module-b { /* Copies everything from .module-a */ .module-a(); border: 1px solid red; }
.module-a { /* A bunch of stuff */ } .module-b { /* A bunch of stuff */ border: 1px solid red; }

Sass

.module-a { /* A bunch of stuff */ } .module-b { /* extends selectors, not code */ @extend .module-a; border: 1px solid red; }
.module-a, .module-b { /* A bunch of stuff */ } .module-b { border: 1px solid red; }

Variable scoping

Less

@color: black; .scoped { @color: white; color: @color; } .unscoped { // Black, as it should be color: @color; }

Sass

$color: black; .scoped { $color: white; color: $color; } .unscoped { // White, as it was overwritten color: $color; }

And then Less gets left behind

@content

A mixin include can now accept a block of content. The style block will be passed to the mixin and can be placed at the point @content is used

Sass

@mixin iphone { @media (max-width: 480px) { @content; } } @include iphone { body { color: red } }

CSS

@media (max-width: 480px) { body { color: red } }

Directive Interpolation

You can now output values of variables into standard CSS directives such a @font-face and @media

This allows for complicated mixins in a much simpler syntax

$media: screen; $feature: -webkit-min-device-pixel-ratio; $value: 1.5; @media #{$media} and ($feature: $value) { ... }

Even more features in Sass 3.2

  • Mixins can be defined in a nested context i.e. within @media queries
  • CSS4 selectors are now supported
  • min() and max() math functions
  • Placeholder selectors: an extension to @extend and mixins

Compass

  • All of those basic Less mixins rolled into one
  • Sprites, vendor prefixes, grids, gradients, it's all there
  • Also things not possible with less: extra math functions like log(), sqrt(), pow(), plus tint() and shade()

Wait! It's not over

Less 1.3.1 came out 3 days ago

A big (little) update

  • Much improved rule engine: Bootstrap compiled went from 6221 lines to 5270. ~18% improvement
  • New colour functions: tint, shade, multiply, screen, overlay, hardlight, difference, exclusion, average, negation, softlight, red, green, blue, contrast
  • Directive Interpolation, just like Sass 3.2
  • Supports new value units dpi, vmin, vm, dppx and dpcm
  • Still doesn't resolve the extensible class issue
